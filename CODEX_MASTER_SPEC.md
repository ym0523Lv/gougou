## 1. 产品定义

### 1.1 一句话

> 勾勾是一张私密、可回看的“人生日历”：用户自己决定哪一天值得留下一个勾；想写时，再用文字和照片留下当时。

它不是任务清单，也不只是换了日历外壳的普通日记。它把三个动作连成一个闭环：

1. **看见时间正在经过。**
2. **亲自判断这一天是否值得一个勾。**
3. **在以后通过勾、文字和照片重新遇见当时的自己。**

### 1.2 要解决的问题

大多数人的日常高度相似。一个月或一年结束后，时间常常只剩下模糊的整体感：

- 不知道这些日子具体发生过什么；
- 想写日记，但长篇记录的门槛太高；
- 照片散落在相册里，缺少当时的想法；
- 私密内容不敢放进会联网、会同步或边界不明的产品；
- 换手机、应用停运或格式封闭后，担心多年记录被困住或丢失。

勾勾用一个极低摩擦、但需要主动判断的“勾”制造时间张力；用可选的文字和照片保存证据；用月历、年历和回忆入口把人生重新呈现出来。

### 1.3 产品北极星

> 让相似的日子不再悄悄消失。

更完整的内部表述：

> 时间带来压力，意义由用户定义，产品只负责忠实保存并在合适的时候归还记忆。

### 1.4 “有张力、无惩罚”

勾勾不追求“完全没有压力”。如果空白与填满没有区别，勾就失去价值。需要保留的是对时间流逝的感知，禁止的是替用户定罪。

| 保留的张力 | 禁止的惩罚 |
| --- | --- |
| 年历中的空白自然可见 | 把空白标红或称为失败 |
| 晚间询问“今天，值得一个勾吗？” | “你今天还没完成”“断签了” |
| 展示“今年留下了 N 个勾” | 完成率、漏签数、排行榜 |
| 用户主动给今天下判断 | 自动打勾、系统替用户定义意义 |
| 可以取消误勾并修正记录 | 失去连续天数、扣分或制造羞耻 |

休息、发呆、拒绝一件事、认真吃一顿饭，都可能是用户认为有意义的事情。产品永远不把“有意义”等同于生产力。

### 1.5 品牌表达分层

对外传播可以直接、有力量：

- “给没白过的今天，留一个勾。”
- “把日子勾出来，把人生留下来。”
- “一年以后，你看到的不只是一串日期。”

应用内部要更安静：

- 今日提示：“今天，值得一个勾吗？”
- 写作提示：“想记下为什么吗？”
- 年度回看：“这一年，你留下了 216 个勾。”

“这一天没白过”是品牌主张，不应成为每次打开都出现的审判语。

## 2. 用户、场景与产品边界

### 2.1 第一用户

第一用户是产品创建者本人。达到 `Safe Dogfood` 后，再把勾勾作为真实主力日记持续使用；随后逐步邀请可信的小范围用户，最后公开上架。

Dogfood 不是降低质量要求。真实私密数据一旦写入，数据保留、升级和恢复从第一天起就是正式约束。

### 2.2 目标用户

- 觉得时间过得太快，希望看见自己一年如何生活的人；
- 写不了长日记，但愿意每天做一次简短回望的人；
- 想把难以启齿的想法、私人照片和日常碎片放在一个可信空间的人；
- 不喜欢社交展示、效率打卡和算法评价的人；
- 重视本地保存、迁移、导出和长期可读性的人。

### 2.3 核心使用场景

1. **当天确认**：睡前看见今天，决定是否给它一个勾。
2. **随手留下**：用一句话、一段文字或一张照片记住不同之处。
3. **补记过去**：在记忆仍清楚时补写或修正过去的日期，但不称“补卡”。
4. **月年回看**：看到勾逐渐填满；从年历选择月份，再在月历选择具体日期回到当时。
5. **私密倾诉**：记录不愿出现在普通相册、云笔记或社交网络里的内容。
6. **换机与自救**：无需账号，把完整记录安全迁移到新设备，并能导出为普通文件。

### 2.4 明确不是

- 不是待办事项、习惯追踪或生产力评分工具；
- 不是强制长文、公开动态或精修内容的写作社区；
- 不是以连续签到、勋章、积分、广告为增长手段的打卡产品；
- 不是默认联网或把数据锁进私有格式的云服务；
- v1 不是社交产品，也不包含漂流瓶。

## 3. 核心模型与行为规则

### 3.1 双层记录

每个民用日期只有一条稀疏记录，包含两个互相独立的层次：

1. **勾**
   - 用户亲自做出的结论：“我愿意记住这一天。”
   - 必须显式点击，不因写字、拍照或打开应用自动产生。
   - 可以单独存在。

2. **内容**
   - 可选的文字和图片，用来保存当时发生的事、感受或原因。
   - 可以在未打勾时先写下。
   - 可以解释一个勾，也可以独立存在。
   - 写作不应变成获得勾的条件。

### 3.2 日期状态

| 状态 | 含义 | 月历表现 | 年历表现 |
| --- | --- | --- | --- |
| 空白 | 没有勾，也没有内容 | 仅显示日期 | 中性日期点 |
| 只有勾 | 用户确认过这一天 | 明确勾形 | 勾标记 |
| 只有内容 | 留下了内容，尚未给出结论 | 独立的纸页点或短线 | 内容标记 |
| 勾与内容 | 确认且有可回看的记忆 | 勾形与内容标记同时出现 | 组合标记 |

月历和年历都必须靠形状区分四种状态，不能只靠颜色。年历日期本身是非交互视觉单元，由月份卡统一提供可访问摘要。

### 3.3 日期规则

- `entry_date` 是用户当前时区下的民用日期 `YYYY-MM-DD`，不是 UTC 时刻。
- 今天和过去日期允许打勾、取消勾选、写作和修改。
- 未来日期允许浏览，不允许打勾或写日记。
- 打开编辑器后锁定目标日期；跨零点保存仍写回原日期。
- 应用跨午夜、切换时区或从长时间后台恢复时，必须重新计算“今天”。
- 空白日保留为空白，不自动补齐，也不显示负面状态。

### 3.4 正反馈机制

正反馈来自累积出来的人生图景，而不是外部奖品：

- 打勾时提供克制的动画和可关闭触感；
- 月历呈现本月的形状；
- 年历呈现十二个月的密度；
- 只展示正向事实，例如“今年留下 N 个勾”；
- 年历点击月份卡进入月历，再选择有标记的日期回到当天；有内容时优先进入阅读状态；
- Post-v1 可加入“往年今日”和温和月度回顾。

v1 不显示连续天数、最长连续、完成率、漏签数量、火焰、勋章或排行榜。年历本身已经同时提供了压力和奖励，再加一套惩罚型指标会诱导用户为不断签而勾，而不是为有意义而勾。

## 4. 核心用户闭环

```text
启动
  └─ 隐私门（若启用）
       └─ 月历 / 今天
            ├─ 显式打勾
            ├─ 写文字或插入照片
            ├─ 切换年历 → 选择月份 → 月历选日 → 回看
            └─ 设置
                 ├─ 提醒
                 ├─ 隐私
                 ├─ 数据迁移与导出
                 └─ 外观与辅助功能
```

每日闭环：

```text
时间提醒或主动打开 → 思考今天 → 显式打勾 → 可选留下原因
```

长期闭环：

```text
勾逐日累积 → 月/年视图形成可见人生 → 年历选月、月历选日 → 唤起记忆 → 更愿意继续留下今天
```

信任闭环：

```text
应用默认不主动上传 → 私密内容敢于写入 → 可验证备份/迁移 → 用户愿意长期托付
```

## 5. 信息架构与功能模块

### 5.1 启动与隐私门

目标：

- 健康检查、设置加载和必要迁移完成前，不展示日记摘要。
- 启用隐私锁后，冷启动先验证系统生物识别或设备凭据。
- 只有系统认证能力可用且完成一次成功验证后，才能开启隐私锁。
- 隐私锁必须允许设备凭据回退，不能只依赖可能变化的单枚生物特征。
- 已开启后若认证服务暂时不可用，保持锁定并提供重试；若设备凭据被移除，保持锁定并引导重新设置设备凭据，不能静默关闭锁。
- 用户无法恢复系统认证时，只能使用已验证的便携备份恢复到新安装；因此启用隐私锁前必须解释这一边界。
- 进入后台时立即用无内容遮罩覆盖系统任务切换预览；超过设定时长后要求重新验证。
- 解锁失败、取消或系统暂时锁定时，内容继续不可见。
- 未保存草稿不能因锁屏组件切换而丢失。

首启只用一张可关闭的说明，不做多页 onboarding：

> 今天，想留下一个勾吗？什么值得记住，由你决定。想写时，再留下文字或照片。

随后明确告知：应用默认不主动上传、系统与手动备份边界，以及丢失设备前需要做什么。

### 5.2 月历首页

月历是每日操作中心，默认定位今天。

必须具备：

- 周一为一周首日；
- 月份切换和快速回到今天；
- 首次进入当前月默认选择今天；进入历史月或未来月时默认不选择日期，用户选定具体日期后才显示日期动作；
- 从编辑器返回时可恢复刚刚明确选择的日期，不能因切月沿用旧日号并产生误操作；
- 今日、当前选择、勾、内容四种语义互不覆盖；
- 选择日期后，底部显示完整日期和该日状态；
- 主动作是显式勾选，次动作是写作；
- 历史日期文案使用具体日期，不再统一写“和今天打个招呼”；
- 未来日期动作禁用并解释原因；
- 所有日期提供自然中文的读屏名称。

### 5.3 年历

年历不是后续装饰，而是核心价值在长时间尺度上的呈现。

v1 必须具备：

- 冷启动和普通回到首页时默认显示月历，年历不取代每日入口；
- 从月历一层可达的“月 / 年”切换；
- 手机上纵向浏览十二个紧凑月份；
- 每个日期是非交互视觉单元，确定性地表达勾与内容两种标记；
- 每个月份卡是至少 `48dp` 的操作目标，点击后进入该月月历；
- 读屏只聚焦月份卡，并读出“7 月，8 个勾，3 天有内容”等摘要，不制造约 500 个逐日焦点；
- 当前年显示“今年留下 N 个勾”，历史年显示“2025 年留下 N 个勾”，未来年只显示年份；
- 切换年份并能快速回到今年；
- 不显示完成率和未完成数量。

视觉要求：

- 标准手机宽度优先使用两列月份卡；窄屏或大字体自动退为一列；
- 月份卡使用有呼吸感的留白、克制边框和统一圆角，不做密集数据表；
- 勾形在十二个月中形成清晰、稳定的视觉节奏，内容标记退居次级；
- 月份标题、星期和日期保持一致基线，空白日不使用灰色叉号或失败色；
- 浅色、深色、360/390/412dp 宽度和 200% 字体都需要视觉验收截图。

### 5.4 日记阅读与编辑

日期就是标题，不增加普通笔记式“标题”输入框。

交互原则：

- 空记录从“写几句”进入后直接得到一整块连续画布。
- 有内容的历史日期优先以阅读状态打开，并提供明确、可访问的“编辑”按钮；点击正文空白区域可以是辅助入口，但不是唯一入口。
- 今天或刚创建的记录可以直接进入编辑状态。
- 富文本是用户界面，Markdown 是内部存储和可移植格式；不提供编辑/预览双模式。
- 软键盘出现时，单行格式栏紧贴键盘上方；键盘收起时可隐藏常用栏，但编辑状态始终保留可访问的“更多格式”入口。
- 高频入口控制在图片、粗体、小标题、列表；引用、待办、撤销/重做等进入可横向访问区域或“更多”面板。
- 待办只是可选文本格式，不作为产品主语义，也不进入月历统计。
- 正文使用系统可缩放字体，目标基准约 `17sp`、行高 `1.65–1.75`、左右留白约 `20dp`。
- 主要触控目标不小于 `48dp`；小屏、横屏、200% 字体和中文输入法必须可用。
- Android 返回键在键盘展开时先收起键盘；再次返回才离开编辑页。

保存原则：

- 无手动保存按钮。
- 输入先形成本机可恢复草稿，再以约 1500ms 防抖写入 SQLite。
- 正常“已保存”短暂出现后淡出；保存中和失败持续可见。
- 返回、切后台、隐私锁和页面卸载时触发尽力 flush；可靠性由短防抖、独立恢复草稿和下次启动恢复共同保障，不承诺 OS 强杀前绝对提交完成。
- 保存失败时保留草稿并可重试，不能让图片失败冒充正文未保存。
- revision 冲突必须保留本地草稿并提供恢复路径，不能用旧 revision 无限重试。
- 编辑器加载完成前不可输入，防止异步载入覆盖快速输入。

图片原则：

- 使用系统 Photo Picker 或文件选择器，只访问用户选中的项目，不申请整个图库权限。
- v1 固定提供“从系统相册选择”；返回后插入原光标位置。
- 直接拍照在完成临时文件、取消、进程恢复、EXIF 方向和权限设计后进入后续版本。
- 私密照片保存在应用沙盒，不自动进入系统相册。
- 插入后立即显示本地预览和处理状态；失败不留下无效占位。
- v1 单列按正文宽度显示；多图拼贴、滤镜和完整相册能力后置。
- 原图、缩略图和引用必须拥有明确生命周期，不能因另一篇日记保存而误删待保存图片。

Markdown 兼容合同：

- v1 无损支持段落、二至三级小标题、粗体、斜体、无序列表、有序列表、待办列表、引用、代码块、图片和换行。
- 导入的一级标题必须保留其 Markdown 语义，但编辑器不把它作为常用入口。
- 图片在 Markdown 中只保存受控相对路径，不保存 `file://`、任意绝对路径或 base64。
- 不支持的语法必须保留原文、以安全只读方式展示或在导入前明确拒绝，不能静默改写。
- 每次 Tiptap/Markdown 升级必须通过固定金样的“读取 → 编辑 → 保存 → 重开”往返测试。

### 5.5 回忆

v1 的回忆能力是“从年月视图点回任意日期”，不另做复杂时间线。

Post-v1 候选：

- 往年今日；
- 本月随机三天；
- 本地搜索；
- 按有照片、有文字、有勾筛选；
- 单日 Markdown/PDF 分享。

回忆入口只使用本地数据，不为了推荐建立云端画像。

### 5.6 晚间提醒

- 默认关闭，用户主动开启后才请求通知权限。
- v1 只承诺“大约时间”的本地提醒，删除“尽量准时”和精确闹钟特殊权限。
- 当天已打勾则取消当日提醒。
- 统一文案：“今天，值得一个勾吗？”
- 通知不得包含日记正文、照片标题或敏感摘要。
- 可修改时间、暂停一周；静默日作为高级选项折叠。
- 权限拒绝后不重复骚扰，只在设置页展示恢复入口。
- Android 厂商后台限制只做真实说明和用户主动入口，不承诺所有设备绝对送达。

### 5.7 设置

设置按用户目的组织，而不是按技术名词堆叠：

1. 提醒；
2. 隐私；
3. 数据与迁移；
4. 外观与辅助功能；
5. 关于与版本。

设置必须区分“用户想要的状态”和“系统实际允许的状态”。持久化失败时回滚界面或明确显示尚未保存，不能展示虚假成功。

## 6. 私密性与安全承诺

### 6.1 产品原则

私密性可以成为用户选择勾勾的最强理由，但不能只是增长文案。越是鼓励用户写难以启齿的心事和保存私密照片，越必须把承诺写成可验证的工程约束。

公开宣传只允许描述已经实现并验证的能力。

### 6.2 默认数据行为

Android 公开版必须满足：

- 不需要账号；
- 不包含广告或远程分析 SDK；
- 不主动上传日记、图片、日期或使用行为；
- release Manifest 不包含业务不需要的 `INTERNET` 权限；
- 显式关闭 Android Auto Backup，避免数据库和图片被系统默认上传；
- 云备份默认关闭，且 v1 不实现应用自有云；
- 日记数据库、资源和临时文件只位于应用私有目录；
- 原生提醒缓存不含正文、图片或数据库路径；
- 日志和错误信息不记录正文、图片内容、恢复密语或完整私有路径。

iOS 公开版开始前必须制定对等规则并独立验证系统备份排除、任务切换遮罩、LocalAuthentication、通知和数据目录行为。

### 6.3 威胁模型

v1 要保护：

- 他人拿到已解锁手机后的随手查看；
- 系统任务切换预览泄露；
- 普通应用跨沙盒访问；
- Android 系统默认备份把私密目录上传到用户云空间；
- 备份文件丢失或被复制后的内容暴露；
- 应用自身或第三方 SDK静默上传；
- 损坏、伪造或不兼容备份破坏本地数据。

v1 不承诺抵抗：

- 已 root、越狱或被恶意系统控制的设备；
- 绕过操作系统保护后直接取得应用私有目录的离线攻击；
- 用户主动导出的未加密可读文件再次泄露；
- 用户遗失独立恢复密语后的服务端找回；
- 物理取证级攻击。

这些边界必须出现在隐私说明中，不能使用“绝对安全”。

### 6.4 设备内二次加密决策门

Android 的应用沙盒、设备实际提供的系统存储保护、应用锁、即时遮罩、关闭 Auto Backup 和加密便携备份，是公开 v1 的强制基线。当前 `minSdk 24` 不能保证每台设备都启用了 FBE，因此公开文案只能说“继承设备实际提供的系统加密”，不能宣称应用强制所有设备具备统一的文件级加密。

应用层数据库/图片二次加密不是在 PRD 中先验决定的 v1 阻塞项。核心闭环达到 Safe Dogfood 后，必须完成一次有时限的安全与工程验证，并二选一：

1. **保持基线承诺**：对外只承诺“本地保存、应用锁、系统沙盒、应用不主动上传、便携备份加密”，明确设备内数据继承设备实际的系统保护，不宣传“保险箱级设备内加密”。
2. **扩大威胁模型和市场承诺**：若要抵抗“应用私有目录被离线复制后读取”，则应用层加密成为公开版阻塞项。

技术验证必须输出：

- SQLCipher 全库加密与“正文/图片应用层 AEAD”两条方案的构建、迁移、恢复、体积和性能数据；
- 每条方案仍会暴露的元数据表。日期、勾、字数、更新时间、设置和资源关系本身都视为敏感；若保留明文，只能宣传“正文和图片加密”，不能宣传“数据库已加密”；
- 平台密钥生命周期：创建、认证绑定、进程后台释放、增删指纹、修改锁屏凭据、密钥失效、轮换、升级、D2D 只恢复密文、清除数据和卸载；
- 受控资产协议如何在内存中认证解密，禁止为了显示图片把长期明文写回临时目录；
- 低空间、迁移中断、进程被杀、密钥失效和旧版本升级的恢复结果。

若选择应用层加密：

- 使用经过审计的 AEAD/KDF 实现、固定算法版本和测试向量，不自创密码学；
- 随机主密钥由平台 `KeyProvider` 保护；Android 使用 Keystore，未来 iOS 使用 Keychain/Secure Enclave 对等能力；
- Rust 通过最小原生桥接获取解封能力，明文主密钥不写入磁盘；
- 数据库与图片使用分离的用途密钥，每个密文有独立 nonce 和完整性认证；
- 设备密钥永不进入便携备份；备份在新设备解密后，用新设备密钥重新保护本机数据；
- 先完成加密容器和可验证内部回滚快照，再执行 `staging → 校验 → fsync → 原子切换 → 延迟删除旧代` 的明文迁移。

选型完成前不得对外宣称设备内数据已做应用层加密。

### 6.5 屏幕与通知

- 启用隐私锁后，Android 默认遮挡最近任务缩略图，并启用防截屏；用户可在理解影响后关闭防截屏。
- 应用进入后台立即盖住内容，30 秒只决定是否需要重新认证，不决定是否遮挡。
- 通知只出现通用邀请，不出现私人文本。
- 分享和可读导出必须经过明确用户动作，离开私密边界前再次提示。
- 设置提供“删除全部本机数据”入口并二次确认；应用不能承诺远程删除用户已复制到外部位置的导出和备份。
- 若启用应用层加密，可通过销毁设备主密钥实现快速密码学擦除，但不能承诺闪存物理安全擦除。

## 7. 数据保留、备份、导出与换机

### 7.1 数据所有权

用户必须同时拥有两种退出路径：

1. **可完整恢复的备份**：用于换机、灾难恢复和版本迁移。
2. **人类可读的导出**：即使勾勾停止维护，也能用普通工具阅读。

“能导出 ZIP”不等于完成数据主权。需要证明导出的文件可在全新安装中恢复，并长期保留格式说明和兼容测试样本。

### 7.2 默认完整备份

公开 v1 使用版本化的 `.gougou-backup` 容器：

- 默认加密；
- v2 基线使用应用生成、至少含 128 位随机熵的恢复密语，不接受可离线猜测的短自定义密码；
- 首次显示恢复密语时要求用户再次输入指定部分，并执行一次真实解密验证；应用不保存或上传恢复密语，遗失后无法由开发者找回；
- 每个备份生成独立的 256 位随机数据密钥；使用参数有版本且经过设备基准的 Argon2id 从恢复密语导出密钥封装密钥；
- 数据密钥只以 XChaCha20-Poly1305 认证密文形式被封装，manifest 与内容块再通过 HKDF-SHA-256 分离用途密钥；
- 所有 nonce 由系统 CSPRNG 生成，同一密钥下绝不复用；算法、参数和 nonce 规则属于格式合同，改变时升级容器版本；
- 未解密 header 只包含 magic、容器版本、受限的 KDF 参数与 salt、分块布局所需信息；
- 应用版本、创建时间、日期范围、数量、大小、资源清单和内容完整性信息全部位于加密且认证的 manifest；
- header 作为 AAD 绑定；认证覆盖块顺序、总块数和结束标记，拒绝重排、截断和拼接；
- 导入端限制不可信 KDF 参数、块数和声明大小，防止内存/CPU DoS；
- 格式冻结前必须经过独立安全评审，并提供固定测试向量和跨版本解析测试；
- 包含勾、正文、设置、图片及引用关系；
- 导出采用流式读写，不把全部图片同时放入内存；
- 文件名包含日期、时间和短随机后缀，避免覆盖。

当前明文 ZIP 备份格式继续作为 v1 格式迁移输入，但不能作为公开版默认安全备份。

### 7.3 人类可读导出

用户可显式选择未加密导出：

```text
Gougou-Export/
  calendar.json
  entries/
    2026/
      2026-07-23.md
  assets/
    2026-07-23/
      ...
```

可选提供 PDF，Markdown + 原图是最低永久承诺。导出前明确提示：

> 这是未加密文件。能访问保存位置的人可能读取其中的文字和照片。

### 7.4 导入与换机流程

标准换机流程：

```text
旧手机导出加密备份
  → 用户把文件交给系统文件提供器、线缆或自己的存储介质
  → 新手机安装勾勾
  → 选择备份并输入恢复密语
  → 预检摘要
  → 恢复
  → 校验条目、图片和设置
```

导入必须：

- 在写入前检查格式版本、KDF、认证标签、日期、设置值、数量、解压大小和资源引用；
- 恢复密语验证成功后才展示日期范围、勾数量、内容数量、图片数量、备份版本和所需空间；
- 默认“合并日记，保留本机设置”；
- 合并时只自动导入本机不存在的日期；同一日期内容不同即列为冲突，默认保留本机，不用跨设备 `updated_at` 猜测先后；
- 冲突界面允许逐日选择“保留本机”或“使用备份”，并提供批量全部保留本机/全部使用备份；
- “替换全部”明确替换日记并要求二次确认；
- 主题、减少动画和触感可以随“替换全部”恢复；提醒只恢复时间等用户意图并等待新设备重新授权；
- 隐私锁、系统权限、原生闹钟缓存、平台密钥和生物识别状态永不从旧设备直接恢复；
- 恢复先在新数据世代目录中完整生成 SQLite、资产和设置，并写入包含旧/新世代与阶段的恢复日志；
- staging 完成后校验引用、大小和数据库健康，执行文件与目录 `fsync`，再关闭旧连接并原子切换一个很小的“当前世代”指针；
- 启动时读取恢复日志：指针切换前的中断删除新世代并继续旧数据，指针切换后的中断完成新世代校验并保留旧世代供回滚；
- 旧世代只在新世代成功启动并通过延迟复核后删除；
- 预检按旧数据、staging 和新资源的最坏情况估算空间；
- 提交前失败时原数据库、图片、设置和提醒不变；提交后数据库世代是唯一事实来源，不尝试用提醒失败回滚用户数据；
- 提交后以幂等方式刷新主题、月历、年历并从新数据库重建提醒；提醒协调失败时显示降级状态、保留新数据并在启动/前台恢复时重试；
- 提供进度；只允许在 staging/验证阶段取消，进入最终原子切换后不再提供取消；
- 防止同一导入令牌重复使用。

### 7.5 云备份

云备份保留为 Post-v1 可选能力：

- 默认关闭；
- 用户明确选择保存位置或提供商；
- 上传前已在设备端加密，提供商不能读取正文和图片；
- 第一版只做用户主动通过系统文件提供器“保存到”云盘，本质仍是手动导出；
- 定时后台备份需要另行设计持久 URI 权限、后台调度、失败重试和供应商行为，不与手动导出混称；
- 不先建设账号和自有服务器；
- 明确展示最后成功时间、包含范围、恢复密钥状态和失败原因；
- 关闭后不继续后台上传；
- 未完成跨设备冲突模型前只做“备份/恢复”，不称“同步”。

## 8. 漂流瓶与陌生人沟通

漂流瓶可能提供“有人理解这一刻”的额外价值，但它会同时引入联网、服务端存储、内容审核、骚扰、未成年人保护、举报封禁、反垃圾和隐私误分享。它改变的不是一个功能模块，而是产品的信任边界。

因此：

- 不进入 Android v1；
- 在私密日记、备份和公开版信任建立后，作为后续独立实验评估；
- 最优先考虑独立应用或严格隔离的可选模块，避免私密产品一安装就获得网络能力；
- 不能直接把现有日记“一键扔进瓶子”；
- 用户必须从空白页重新撰写要公开的内容；
- 第一版只允许文本，不允许照片、位置、联系人、设备标识和私密日记元数据；
- 必须有明确保留期限、删除机制、举报、屏蔽、速率限制、审核和年龄政策；
- 社交数据不得进入私密日记备份，也不得读取私密数据库内容。

在完成安全、法务、审核成本和最小实验设计前，不创建服务端。

## 9. 视觉与交互设计系统

### 9.1 品牌气质

- 温暖，但不幼稚；
- 私密，但不阴暗；
- 有时间重量，但不制造绩效感；
- 像一本长期使用后越来越有价值的私人册子，而不是通用后台管理页。

建议视觉语言：

- 暖白纸张色作为画布；
- 低饱和墨色作为正文；
- 单一温和的玉色或青绿色作为勾与主要动作；
- 危险操作单独使用克制的危险色；
- 勾的品牌图形更像手写确认或勾勾手指的约定，不使用普通任务复选框；
- 动画只服务于状态确认，不使用烟花、火焰和夸张庆祝。

### 9.2 语义设计 Token

保留 Tailwind，但组件颜色只使用语义变量，至少包括：

```text
--surface-canvas
--surface-card
--surface-raised
--text-primary
--text-secondary
--text-muted
--border-subtle
--accent
--accent-on
--warning
--danger
--focus-ring
```

不再通过覆盖具体 Tailwind 色类实现暗色模式。

### 9.3 屏幕原则

- 每屏一个主动作；
- 内容优先，导航和格式工具按需出现；
- 手机采用单栏；桌面双栏不阻塞 Android 首发；
- 小屏和大字体允许滚动，不把按钮压缩到不可用；
- 安全区、系统栏、IME、横屏和折叠屏窗口变化视为一等布局输入；
- 所有状态有文本或语义，不只依赖颜色、位置和动画；
- `lang="zh-CN"`、应用标题、图标、启动主题和商店素材必须去除模板品牌。

## 11. 技术栈决策

### 11.1 保留现有栈

继续使用：

- Tauri 2；
- React 18 + TypeScript；
- Tailwind CSS 4 + 语义 CSS Token；
- Rust；
- rusqlite / SQLite；
- Tiptap 3；
- Kotlin/Swift Tauri 移动插件。

原因：

- 现有 Android 核心链路已经投入较多实现和真机验证；
- SQLite 适合本地、稀疏、按日期访问的数据；
- Rust 统一拥有数据库和文件边界是正确的安全结构；
- 已识别的主要风险是产品语义、状态机、数据一致性和发布工程，不是框架能力不足；
- 换 Flutter、React Native、原生或 ORM 会扩大迁移面，不能直接解决数据安全阻塞项。

在取得 release 包体、PSS、启动时间和编辑体验基线前，不重写技术栈。只有客观门槛无法达到且局部优化无效时，才提交带测量数据的重写提案。

### 11.2 技术约束

- Tiptap Markdown 仍标记为 Beta：固定版本并建立 Markdown 往返金样测试，不跟随大版本自动升级。
- 图片处理、加密和 ZIP 等耗时 command 必须移出 UI 主线程。
- 在改变编辑器和备份状态机前先建立最小自动测试。
- iOS 没有真实编译和真机证据前不得宣称支持。

```text
  android/
  ios/
  src/

docs/
  validation/
  archive/
```

规则：

- `lib.rs` 只负责 Tauri builder、state、插件和 command 注册；
- command 做参数边界和协调，业务逻辑在 service/domain；
- Rust 是 SQLite 和私密资源的唯一所有者；
- 前端通过单一类型化 IPC gateway 调用 Rust，不散落命令字符串；
- React 只持有页面状态和短期草稿，不引入全局状态库；
- 单一导航层统一 Android 返回、通知目标和未来 iOS 手势；
- 先建立测试和模块边界；后续只在对应职责被修改时逐步迁移，禁止一次性搬空整个文件。

## 13. 数据与后端方案

### 13.1 SQLite

- 保留单 SQLite 连接和明确串行写入，暂不引入连接池或 ORM。
- 使用 `PRAGMA user_version` 逐版本迁移，禁止发布后删除重建。
- 每个迁移提供旧版本 fixture、成功升级和失败回滚测试。
- SQLite schema、设备内加密存储格式和便携备份格式使用三套独立版本号，不能互相代替。
- 前端只消费“数据库已就绪 / 需要升级 / 版本过新”等兼容状态，不硬编码要求某一个 schema 数字。
- 设置值在导入预检阶段完整解析，不能只验证键名。
- `toggle_tick` 只拥有勾字段；`save_entry` 只拥有内容、字数、revision 和引用。
- 月/年查询只返回摘要，不返回全部正文。

### 13.2 资产生命周期

下一次 schema 迁移增加受控资产元数据，至少表达：

```text
asset_name
media_type
byte_size
integrity_value
state           # pending | ready
created_at
```

未启用设备内二次加密时，`integrity_value` 可以是明文内容的 SHA-256，用于损坏检查。若启用 AEAD，内容完整性由认证标签保证；不得把私密图片的可比对明文哈希作为公开元数据，额外哈希必须明确是密文哈希或 keyed digest。

流程：

1. 导入图片为 `pending`；
2. 日记保存事务建立引用并转为 `ready`；
3. 取消编辑后只回收过期 `pending`；
4. 删除引用后只检查本次移除的资产；
5. 启动时执行恢复性 GC；
6. 保存时验证资源真实存在，不能只验证文件名语法。

### 13.3 并发与耗时任务

- 小型 SQLite 查询可保持短同步临界区。
- 图片解码、缩放、加密、哈希、ZIP、导入导出和大文件复制使用 async command 外壳，并进入 `spawn_blocking` 或等效阻塞线程池。
- 不持有数据库锁跨越文件 I/O 或 `await`。
- 备份和资产操作使用明确的互斥与固定锁顺序。
- 导出采用流式哈希和流式压缩，导出限制与导入限制保持对称。

### 13.4 错误

- 内部错误保留 source 供本机诊断；
- 前端只收到稳定错误码和不含隐私内容的说明；
- 加载、正文保存、图片处理、备份和系统权限使用独立状态，不能互相冒充；
- 任何“成功”都必须对应已提交、可重新读取的事实。

## 14. 前端状态与可靠性方案

### 14.1 编辑器状态机

至少区分：

```text
load: idle | loading | ready | failed
draft: clean | dirty | flushing
save: saved | failed | conflict
image: idle | picking | processing | failed
```

约束：

- `loading` 时正文不可编辑；
- 每个保存请求绑定日期、revision 和请求序号；
- 旧响应不得覆盖新草稿；
- revision conflict 重新读取已保存版本，同时把本地内容保存为冲突草稿；
- 冲突界面明确显示“这一天在别处发生了变化”，提供“保留已保存版本”和“用我的草稿替换”两个动作；替换必须基于刚读取的新 revision 发出一次新的显式保存；
- 用户作出选择前，两个版本都不能被 GC、锁屏或重新进入页面删除；
- 锁屏只覆盖 UI，不销毁尚未安全持久化的状态；
- 图片失败不改变正文保存状态；
- 月份请求乱序时只接受当前月份结果。

### 14.2 草稿恢复

- 正常内容以 SQLite 为权威；
- 未确认草稿需要独立、受控的本地恢复区，并在成功提交后删除；
- 恢复区位于同一应用私有边界并受隐私锁遮挡；若选择设备内二次加密，它必须使用同等加密；
- 冷启动发现草稿时展示日期和恢复选择，不静默覆盖已保存正文。

### 14.3 日期状态

- “今天”在启动、跨午夜定时点、前台恢复和时区变化后重新计算；
- 是否自动选择新今天由当前页面决定：用户正在回看历史时不强制跳走；
- 通知目标作为一次性导航事件处理，同日期重复事件也不能丢失。

## 16. 包体、性能与发布门槛

易变的当前包体和 bundle 数字只记录在 `PROJECT_STATUS.md`。debug universal APK 不是商店下载体积，但任何异常大的调试产物都必须进入组成分析，不能用“debug 本来就大”结束讨论。

### 16.1 先测量

在同一设备、同一提交、同一场景分别记录：

- universal debug APK；
- arm64 release APK；
- release AAB 和 bundletool 估算下载大小；
- 新安装的 code/data/cache；
- 月历、空编辑器、长文、多图、后台恢复的主进程与 WebView renderer 分项及合计 PSS/RSS/USS；
- 至少 10 次真正冷进程启动的 `am start -W` 中位数和 p95；
- 输入到绘制、保存 IPC 和大型备份操作的延迟分布；
- 20 次页面往返和前后台循环后的稳定性。

### 16.2 目标假设

以下数值是第一轮 release 测量前的目标假设，不直接作为 RC 硬门。完成基线后，由产品负责人记录“批准为门槛 / 调整并说明原因 / 暂缓”，附复审日期；批准后不能静默放宽：

- arm64 release APK ≤ 40 MiB；
- AAB 的 arm64 估算下载大小 ≤ 30 MiB；
- 新安装且无用户内容时，应用自身 code + data ≤ 80 MiB；
- 月历静置 60 秒合计 PSS 目标 ≤ 160 MiB；
- 20 次循环后静置 PSS 相对首次增长 ≤ 20 MiB；
- vivo 基准机冷启动 TotalTime 中位数目标 ≤ 800ms；
- Android vitals 与测试矩阵中应用自身 ANR 为 0；
- 输入到绘制和保存 IPC 的目标在第一轮可重复测量后锁定。

### 16.3 优化顺序

1. 构建真正的 arm64 release/AAB 基线；
2. ABI 拆分、release strip、LTO、资源压缩和 R8；
3. 移除模板和未使用依赖；
4. 使用系统中文字体或最小字体子集，避免打包整套 Noto；
5. 懒加载 Tiptap、备份和非首屏模块；
6. 缩略图、图片懒加载和 object URL 释放；
7. 流式备份与耗时 command 后台化；
8. 只有仍无法达标时，才评估编辑器或框架替换。

## 17. 主要风险与处理

| 风险 | 影响 | 处理与发布门 |
| --- | --- | --- |
| 隐私宣传超过真实能力 | 私密内容泄露、信任不可逆 | 按威胁模型逐项验收；未实现不宣传 |
| 恢复密语或设备密钥丢失 | 备份或本机数据不可读 | 密钥生命周期、真实解密验证、明确无法代找 |
| 明文到二次加密迁移失败（若采用） | 现有真实日记损坏 | 加密快照、staging、原子切换、延迟删除和 fixture |
| 备份实现看似成功但无法恢复 | 换机时才发现数据丢失 | 全新安装恢复演练和长期兼容样本 |
| 编辑器加载/保存竞态 | 静默覆盖文字 | 显式状态机、revision、草稿和前端测试 |
| 图片 pending 竞态 | 私密照片丢失或孤儿泄露 | 资产状态表、定向 GC 和事务引用 |
| ZIP/图片同步阻塞主线程 | 卡顿、ANR | async command、阻塞线程池、流式处理 |
| Tiptap Markdown Beta 改写内容 | 格式和文本损坏 | 固定版本、金样往返、升级人工审核 |
| 精确闹钟与 OEM 策略 | 权限复杂、送达承诺失真 | v1 删除精确提醒，只做近似邀请 |
| iOS 源码未经编译 | 虚假的跨平台承诺 | Android 首发；iOS 独立里程碑 |
| 体积与内存过高 | 安装放弃、低端机体验差 | release 门槛和组成报告，不盲目换栈 |
| 系统默认备份先于隐私加固 | 真实 Dogfood 数据可能被系统上传 | A0 立即关闭并验证 Manifest |
| 漂流瓶破坏私密定位 | 误分享、审核与法务风险 | 隔离实验，v1 不联网 |
| 大规模重构同时改行为 | 回归难定位 | 先补测试与安全阻塞项，再按职责小步迁移 |

## 18. 开发顺序

### A0. 立即收紧当前隐私默认值

1. Android 显式关闭 Auto Backup，并同时配置 API 30 及以下与 API 31 及以上的数据提取规则；
2. release 移除 `INTERNET`，仅在确有需要的 debug variant 保留；
3. 为合并后的 debug/release Manifest 建立权限和备份规则快照检查；
4. 文案只承诺“应用不主动上传”；承认部分 OEM 的 D2D 行为不能用单一 Manifest 属性绝对控制。

结束条件：继续 Dogfood 前，当前真实数据不再因应用默认配置进入常规系统云备份。

### A. 数据安全阻塞项

1. 建立最小前端测试框架，并为已知问题先写失败测试；
2. 修复编辑器加载覆盖、状态混淆、revision 冲突、跨午夜 today 和锁屏草稿；
3. 补齐隐私锁启用、认证暂时不可用、设备凭据被移除和恢复路径，禁止永久锁死或静默解锁；
4. 修复备份设置值校验、恢复后全局状态协调和设置合并语义；
5. 在下一次 schema 迁移前移除前端对精确 schema 数字的硬编码；
6. 修复图片 pending 生命周期与资源存在性校验；
7. 把图片和 ZIP 重任务移出主线程；
8. 用现有受控备份路径完成至少一次可恢复演练。

结束条件（`Safe Dogfood`）：无已知静默丢文字、丢图片或破坏数据库的路径，并至少有一份刚刚验证过的恢复备份。达到前，不把唯一重要日记只保存在当前 Alpha 中。

### B. 最小工程基础

1. 建立类型化 IPC gateway；
2. 统一导航、日期工具和 editor/backup 状态机；
3. 建立 CI、迁移 fixture 和错误分层；
4. 先定义模块边界，之后只在相关职责被修改时逐步拆分 Rust `lib.rs`，不做一次性大搬家。

结束条件：A 阶段行为保持不变，后续产品迭代有可回归的最小基础。

### C. 核心产品闭环与持续 Dogfood

1. 实现已经确认的勾语义、过去/未来日期规则和通用提醒文案；
2. 首页保持默认月历，实现好看且可访问的年历月份卡和四种日期状态；
3. 删除精确提醒、特殊权限和相关设置，并迁移既有偏好；
4. 完成历史阅读态、明确编辑入口、键盘工具栏和图片反馈；
5. 达到闭环后开始持续 Safe Dogfood，边用边记录真实摩擦。

结束条件：新用户不需口头解释也能完成“月历 → 勾/内容 → 年历 → 月历回看”，创建者愿意开始把它作为日常工具。

### D. 私密与数据主权

1. 实现后台即时遮罩、Android 防截屏策略和删除全部本机数据；
2. 先完成加密便携备份容器和可验证内部回滚快照；
3. 完成设置页正式导出、预检、冲突选择、替换、人类可读导出和换机；
4. 完成设备内二次加密决策门；
5. 若选择二次加密，再实现平台 KeyProvider、资产协议解密和明文数据迁移；
6. 在全新安装和另一台设备完成恢复演练。

结束条件：可以诚实描述安全边界，应用不主动上传，便携备份丢失后不可读，用户能够完整离开或换机。

### E. 设计、无障碍与发布工程

1. 建立语义设计 Token、正式品牌、图标、首启说明和“关于与版本”；
2. 完成月历/年历/编辑器/设置的浅色、深色、小屏、大字体和核心无障碍；
3. 建立 release 体积、内存、启动、输入和循环稳定性基线；
4. 审批第 16 节目标假设，再按组成报告做字体、懒加载、ABI、Rust 和资源优化；
5. 完成 Android release 权限、签名、升级、备份和可选加密迁移；
6. 完成商店文案、截图、隐私政策和数据安全表，清除模板品牌、开发入口和本机路径。

结束条件：达到已批准的 release 门槛，无数据安全阻塞项，所有公开隐私声明有对应证据。

### F. 最终 Dogfood 与公开测试

1. 创建者在接近发布的同一数据格式上连续使用至少 30 天；
2. 期间至少完成两次完整备份验证和一次全新安装恢复；
3. 记录真正遇到的写作摩擦，不以功能愿望替代证据；
4. 邀请小范围可信用户验证首次理解、月年回看、写作、隐私说明和换机；
5. 修复阻塞问题后进入 Android 公开测试，再决定正式上架。

Dogfood 的成功不是“每天必须打勾”，而是没有数据丢失、用户愿意持续打开、能从年月视图找回具体记忆，并相信自己随时可以安全迁移。

## 19. 产品验证指标

v1 不默认上传行为分析。以下指标通过 Dogfood 日志、受控可用性测试和用户主动反馈验证：

- 首次用户无需解释，能区分“勾”和“写内容”是两个独立动作；
- 能在第一次会话内完成打勾、写一句话、看到已保存并重新打开；
- 能从默认月历切到年历，理解年度图形，再经月份卡找到具体日期；
- 能理解空白不是系统判定的失败，也能理解勾需要自己决定；
- 在手机键盘、长文、图片和系统返回场景中没有内容丢失；
- 能独立导出备份，并在全新安装中完成恢复；
- 能说清数据默认保存在哪里、什么会离开设备、恢复密语丢失意味着什么。

不把“勾的比例”“连续天数”或“每天必须打开”设为产品 KPI。真正的长期信号是用户愿意持续托付记录，并能通过年月视图唤起具体记忆。

## 20. Android v1 完成定义

同时满足以下条件才可称为 Release Candidate：

- “勾”的意义、年月回看和移动写作闭环完整；
- 无已知静默丢文字、丢图片或导入破坏数据的问题；
- 应用沙盒、设备实际系统存储保护边界、应用锁失效恢复、后台遮罩、系统备份规则和“应用不主动上传”均已验证；
- 设备内二次加密决策有测量、有批准、有准确宣传；若采用则迁移和密钥生命周期通过；
