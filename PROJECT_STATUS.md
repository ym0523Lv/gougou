## 3. 已确认值得保留的实现

- Rust 是 SQLite 的唯一业务数据所有者。
- 参数化 SQL、严格日期/月范围和稀疏日期记录。
- 勾与正文各自只修改自己拥有的字段。
- revision 乐观并发基础。
- 应用沙盒、受控资产名、自定义资产协议和图片魔数/尺寸检查。
- `entry_assets` 引用关系和现有备份结构化清单。
- 备份路径穿越、数量、解压大小和 SHA-256 检查。
- 原生提醒插件只管理调度，不直接读写日记数据库。
- React 局部状态 + SQLite 权威数据，不引入 Redux 或 ORM。
- 已完成的 Android 真机证据和 15 个 Rust 测试。

## 4. 数据安全阻塞项

### 4.1 静默数据损失与状态真实性

1. 编辑器在异步载入完成前可输入，后返回内容可能覆盖用户快速输入。
2. 图片导入失败会把正文保存状态标成失败，界面对用户撒谎。
3. revision 冲突只能用旧 revision 原样重试，没有恢复路径。
4. `today` 在应用生命周期中只计算一次，跨午夜/时区后会过期。
5. 隐私锁通过卸载主界面实现，可能销毁仅存在内存中的失败草稿。
6. 月份请求和通知/返回导航缺少统一状态模型。
7. 隐私锁已开启后，认证能力消失或设备凭据被移除时缺少明确恢复流程，存在永久锁死或错误降级风险。

### 4.2 图片一致性

1. 图片导入后立即进入最终 `assets/`，但正文引用要等自动保存。
2. 任意日记保存都可能触发全目录零引用清理，误删另一篇草稿里的待保存图片。
3. 保存只验证资源名语法，不验证文件确实存在。
4. 每次保存扫描全目录会随长期数据增长。

目标：增加 `pending/ready` 资产生命周期、定向 GC 和启动恢复性 GC。

在资产 schema 升级前还必须移除 `src/main.tsx` 对 `schemaVersion === 1` 的硬编码；前端只判断数据库是否就绪或兼容。

### 4.3 备份与恢复

1. 导入只校验设置键，不校验设置值；恶意或损坏备份可让下次启动失败。
2. `merge_newer` 会无条件覆盖设置，但设置没有更新时间。
3. 恢复后前端主题、提醒、隐私锁和原生调度没有统一刷新。
4. 移动端/release 的系统打开和保存后端目前直接返回不支持。
5. 导出把资源整体读入内存；导出限制与导入限制不对称。
6. 同日文件名固定，可能覆盖或失败。
7. 当前格式是明文 ZIP，只能提供损坏检测，不能提供真实性或保密性。

### 4.4 主线程阻塞

20 个现有 `#[tauri::command]` 都是同步命令。图片解码/WebP、ZIP 检查、恢复和导出可能在主线程执行，存在卡顿和 Android ANR 风险。

目标：耗时 command 改为 async 外壳 + 阻塞线程池；SQLite 短事务保持简单。

### 4.5 当前隐私与目标承诺不一致

- SQLite、图片和备份当前均未做应用层加密。
- 隐私锁只是 React/系统认证入口。
- Android Manifest 当前未显式关闭 Auto Backup，系统可能把应用数据纳入用户的系统云备份。
- Android source Manifest 当前有模板 `INTERNET` 权限。
- 后台超过 30 秒才锁定，但系统任务切换快照应立即遮挡。

其中 Auto Backup 和 release `INTERNET` 是当前真实 Dogfood 风险，优先于其他代码重构。设备内应用层二次加密需要单独决策；在这些边界解决并验证前，不能公开宣传“加密保险箱”或“绝不上云”。

### 当前已知体积

- 2026-07-21 的 universal debug APK：`195,113,787 bytes`。
- 用户也观察到约 `187 MB` 的小版本/文件口径。
- `dist` 约 5.3 MiB，其中 Noto Sans SC 字体文件约 4.3 MiB。
- 前端主 JS 约 653 KiB，Tiptap 当前在月历首屏静态加载。

这些都不是 release 商店下载结论。下一轮性能工作必须先产出 arm64 release APK/AAB、安装占用和 PSS 基线。

### 第一轮 release 前的目标假设

- arm64 release APK ≤ 40 MiB；
- AAB arm64 估算下载 ≤ 30 MiB；
- 新安装应用自身 code + data ≤ 80 MiB；
- 月历静置 60 秒合计 PSS ≤ 160 MiB；
- 20 次循环后静置增长 ≤ 20 MiB。

这些数值尚未批准为硬门；取得可重复的 release 基线后，必须记录批准或调整原因。

## 8. 暂不执行但已设决策门

### 设备内加密

在 D 阶段做有时限的 SQLCipher vs 应用层 AEAD 技术验证。Android v1 强制基线是应用沙盒、设备实际提供的系统存储保护、应用锁、即时遮罩、关闭 Auto Backup 和加密便携备份；`minSdk 24` 下不宣称每台设备都具备统一 FBE。是否增加设备内二次加密由扩大后的威胁模型、迁移风险和实测决定。未取得数据前不锁定实现，也不宣传“保险箱级”。

### 云备份

默认关闭，Post-v1 先支持用户主动通过系统文件提供器把加密备份“保存到”云盘；后台自动备份另行设计，不先建设账号或自有服务端。

### 漂流瓶

后续实验，优先考虑独立应用或严格隔离模块。必须先有审核、举报、封禁、保留期限、未成年人和网络隐私方案。

### iOS

不阻塞 Android v1。开始承诺前需要 macOS/Xcode、真实编译、iPhone 真机、VoiceOver、通知、LocalAuthentication、数据目录和备份规则的完整证据。
